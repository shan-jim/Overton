Why this view exists

The Data Systems Overview describes the Centralized Staging Database as the linchpin that joins every M1 quote to its ProfitKey work order. This page shows what that linchpin looks like in practice — how the data gets there (extracts, mapping, jobs, gateway), and what an end user sees when they look up a unified record from a single screen.

The architecture views are anchored to the two source systems in scope for Phase 1 — M1 and ProfitKey; the unified-record walkthrough is anchored to Q1671 (M1) ↔ SO 0271926 (ProfitKey). Pipedrive joins this picture in Phase 3, when production status flows back into deals and parts/material requirements are shareable with vendors — that work has its own viewer when we get there.

This viewer is read-only. The staging database is populated by scheduled extracts; no records are created, duplicated, or edited from this surface. Phase A (this page) shows the architecture and one walked record. Phase B (coming) is team-wide search across the full linked-record history.

Source extracts
Two nightly extractors pull only new or changed rows from each source system — M1 over a SQL Server connection and ProfitKey over ODBC. Production systems are read-only from staging's perspective.

02
Mapping table schema
The staging.linked_records table is the join surface — one row per recognised quote ↔ work-order pair. The link is established by business rule: the M1 quote number is stamped on the ProfitKey work order at release.
Pipeline
Tables staging.linked_records DDL
Join keys⚠ Representative
  • M1 Quote ID ↔ ProfitKey Work Order No. — via business rule (quote number stamped on WO release)
  • Customer ID carried through — qmpCustomerOrgID on the M1 side
  • Estimated hours (qmo) ↔ Actual hours (JOB_HOURS_SUMMARY / LABOR_TICKETS) — via order_no
  • Phase 3 will add a pd_deal_id column for the Pipedrive join

04
Auth & API gateway
The lookup web app sits behind a username/password login. Access flows through the Zero Trust API gateway (per the stakeholder workflow) which exposes only read-only endpoints over the staging tables. Production systems are never reachable from this surface.
Gateway
Layer read-only API expose
Read-only endpoints⚠ Representative
  • GET /quotes/:m1_quote_id — returns header + lines from the M1 mirror
  • GET /work-orders/:order_no — returns header + hours + actuals from the PK mirror
  • GET /linked/:m1_quote_id — returns the unified triple from staging.linked_records
  • GET /actuals/:order_no — returns labor-ticket roll-up for one WO
